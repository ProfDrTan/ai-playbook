Sprint by Sprint: An Empirical Study of AI-Augmented Scrum Pedagogy in Design Thinking Education

Pipeline Architecture, Automation Maturity, and Educator Infrastructure in a Live University Course

Author: Prof Dr Tan Teik Kheong Affiliation: James Cook University Singapore; Stanford University (affiliated) IEEE Senior Member Submitted to: Computers and Education: Artificial Intelligence (Elsevier)  ·  Full Research Article  ·  July 2026

Abstract

This paper presents an empirical study of AI-augmented Scrum pedagogy conducted across eight sprints with five student teams (n = 56) in a live design thinking course (CP3405, TR2 2026). Teams were tasked with building progressively automated market intelligence pipelines using GitHub Actions, Python scripting, and multi-LLM synthesis within a structured Scrum framework. Drawing on an immutable dataset of 492 timestamped GitHub commits, 190 workflow run logs, 25 sprint retrospectives, and five prediction archives, this study proposes the PRISM Automation Maturity Model — a four-stage rubric (Manual → Structured → Automated → Intelligent) for assessing AI pipeline development in educational settings. Results demonstrate measurable differentiation across teams in pipeline reliability, sprint delivery consistency, and calibration accuracy. Three teams achieved fully automated data collection by Sprint 5; one team exhibited automation theatre — the appearance of automation without functional output; and one team built the most architecturally sophisticated modular collector system in the cohort. This paper further proposes a replicable Educator Infrastructure Model comprising a GitHub Actions monitor, Telegram alert system, and PIN-protected live dashboard that provides instructors with real-time visibility into all student pipelines without student knowledge. Findings suggest that when Scrum cadence and AI automation are combined with systematic educator monitoring, design thinking courses can produce demonstrable workforce-ready skills in DevOps, AI/ML engineering, data analysis, and calibrated decision-making — skills that transfer directly to professional contexts.

Keywords: design thinking education; Scrum in higher education; AI-augmented pedagogy; GitHub Actions; automation maturity; educator infrastructure; LLM synthesis; agile learning

1. Introduction

1.1 The Problem

Design thinking education faces a tension that few curricula have resolved. On one side sits the humanistic tradition of empathy-driven ideation, prototyping, and iteration — the intellectual heritage of IDEO, the d.school, and a generation of innovation educators [CITATION — VERIFY: e.g. Brown, T. (2008). Design Thinking. Harvard Business Review; or Dorst, K. (2011). The core of 'design thinking' and its application. Design Studies, 32(6), 521-532]. On the other side sits the accelerating reality of AI-native workplaces, where the skills most valued by employers involve not creative ideation alone, but the disciplined, automated, iterative delivery of working systems on a repeatable cadence.

Most design thinking courses navigate this tension by treating AI as a tool for ideation — a brainstorming partner, a research accelerator, a feedback synthesiser. Few have explored what happens when AI becomes the infrastructure of the course itself: when student teams are required not merely to use AI tools, but to build automated systems that employ multiple AI models, run without human intervention, and produce calibrated outputs that are scored against empirical reality every week.

This paper reports on one such course.

1.2 The Research Context

CP3405 Design Thinking 3 is a final-year undergraduate course run at James Cook University Singapore in Trimester 2, 2026. The course enrolled 56 students organised into five teams across two practicals. Each team was tasked with building and maintaining a market intelligence pipeline — a system that collects financial market data, synthesises analysis through three structured AI agents (Almanac, Macro, Technical), calls multiple Large Language Models via API, applies a human judgment layer (the Human Score Wild Card), and seals a prediction file with a versioned release tag before a weekly Sunday midnight deadline.

The task was not chosen because of financial markets. It was chosen because it is a repeatable, measurable, structured problem that requires every dimension of professional delivery: data engineering, API integration, CI/CD pipelines, Scrum ceremonies, retrospective analysis, and calibration scoring. The market is the data source. Scrum is the discipline. Workforce readiness is the goal.

1.3 Research Questions

This study addresses three research questions:

RQ1: How does automation maturity in student AI pipelines — measured across eight Scrum sprints — correlate with sprint delivery consistency and prediction calibration accuracy?

RQ2: What GitHub Actions workflow architectures and software engineering patterns emerge as best practices when undergraduate student teams build automated AI pipelines under weekly Scrum constraints?

RQ3: What educator infrastructure — monitoring systems, alert mechanisms, and real-time dashboards — enables evidence-based teaching in AI-augmented design thinking courses, and how does instructor visibility into student pipeline health affect pedagogical intervention quality?

1.4 Contributions

This paper makes four original contributions:

  1. The PRISM Automation Maturity Model — a four-stage rubric for assessing student AI pipeline development that is measurable, replicable, and grounded in observable GitHub artefacts.
  1. A taxonomy of student pipeline architectures — derived from analysis of 15 distinct GitHub Actions workflows, five data collection scripts, three CI validation patterns, and one workflow cascade chain across five student teams over six active sprints.
  1. An Educator Infrastructure Model — a complete, deployable technical specification for instructor monitoring, Telegram alerting, and live dashboard systems that provide real-time pipeline visibility without student awareness.
  1. Empirical evidence that automation maturity and Scrum process discipline are measurable through version-controlled artefacts, enabling objective assessment criteria that eliminate self-report bias and grade-gaming.

2. Related Work

2.1 Scrum in Higher Education

The use of Scrum in university courses has been documented since at least 2010, when Mahnic (2010) reported on student perceptions of team-project work under Scrum constraints. Subsequent work by Kropp and Meier (2013) demonstrated that Scrum-based courses produced measurable improvements in students' perceived readiness for professional software engineering. The eduScrum variant, developed by Willy Wijnands and adapted for non-software educational contexts, has been applied in settings ranging from secondary school science to Master's-level digital transformation programmes (Neumann & Baumann, 2021).

What these studies share is a focus on Scrum process — sprint velocity, retrospective quality, stakeholder engagement — as the primary outcome variable. None examines the relationship between Scrum cadence and the automation maturity of student-built systems. This gap is significant: in professional practice, Scrum without automated delivery infrastructure is widely recognised as incomplete [CITATION — VERIFY: e.g. Forsgren, N., Humble, J., & Kim, G. (2018). Accelerate: The Science of Lean Software and DevOps. IT Revolution Press].

2.2 AI Tools in Design Thinking Education

The integration of AI tools into design thinking curricula has accelerated dramatically since 2023. Chiu (2021) proposed a framework for AI literacy in pre-tertiary education; Hwang et al. (2020) reviewed AI-assisted tutoring systems; and a growing body of work in Computers and Education: Artificial Intelligence explores the use of generative AI for feedback, assessment, and learning analytics [CITATION — VERIFY: a specific 2024-2026 CAEAI review article].

However, the predominant model in this literature positions AI as a tool used by students, not as infrastructure built by students. The distinction matters. A student who uses ChatGPT to generate ideas has consumed AI. A student who builds a GitHub Actions workflow that calls the OpenAI API, logs the response as a structured file, and chains that output into a comparison table has engineered with AI. The latter skill is what employers in technology, finance, and consulting roles are hiring for in 2026 [CITATION — VERIFY: e.g. World Economic Forum Future of Jobs Report 2025, or a 2026 LinkedIn/industry skills report — needs a source dated close to submission].

2.3 GitHub as Educational Infrastructure

Version control systems have been used in computer science education for decades, but their use as pedagogical evidence systems — providing instructors with timestamped, objective records of student work patterns — is less documented. Zagalsky et al. (2015) explored GitHub's affordances for collaborative learning; commit activity has been examined as a proxy for engagement in related mining-software-repositories work [CITATION — COULD NOT VERIFY "Bird et al." for this specific claim; closest real matches are Kalliamvakou et al. (2014), The Promises and Perils of Mining GitHub, MSR 2014, or a 2021-2022 paper on git metrics for measuring student code contributions — please confirm the intended source before submission, as I could not locate a Bird et al. paper making this claim]. Neither study examines GitHub Actions as an automation substrate for educational delivery, nor explores the use of the GitHub API as an instructor monitoring tool.

2.4 The Gap This Paper Addresses

No published study combines: (1) Scrum pedagogy, (2) student-built AI pipelines, (3) automated delivery infrastructure, (4) empirical calibration scoring, and (5) real-time educator monitoring in a single design thinking course. This paper addresses that gap.

3. Methodology

3.1 Course Design

CP3405 Design Thinking 3 (TR2 2026) ran for eight weeks, with six active sprint weeks (Sprints 1–5 and Sprint 6) and one study break week. Each sprint followed a standard Scrum cadence: Sprint Planning on Monday, Daily Standups throughout the week, Sprint Review on Monday of the following week, and a Retrospective committed to the repository before the next sprint opened.

The sprint submission protocol required three artefacts before the Sunday midnight deadline:

  • A prediction file (JSON or Markdown) containing directional forecasts for S&P 500 (SPX), Nasdaq 100 (NDX), Russell 2000 (IWM), and sector ETFs, with confidence ratings and invalidation conditions
  • A GitHub release tag with the exact format vWXX (e.g., vW25 for Week 25), serving as the immutable sprint seal
  • A Discord submission post using a standardised template

Ten roles (R1–R10) were defined and assigned each sprint, mapping directly to professional functions: Product Owner (R1), Scrum Master (R2), three Agent leads (R3–R5), Data Collector (R6), Human Score lead (R7), LLM Operator (R8), GitHub Lead (R9), and Calibration lead (R10).

3.2 Data Sources

All data was collected from public GitHub repositories without student involvement in the data collection process. Data sources included:

3.3 The PRISM Automation Maturity Model

Analysis of the five team repositories revealed a consistent four-stage pattern of automation development. This pattern was formalised as the PRISM Automation Maturity Model (Pipeline Readiness Index for Student Market Intelligence):

Stage 1 — Manual: All pipeline steps performed by hand. Data collected by visiting websites, copying values, and committing files manually. LLM synthesis performed by pasting prompts into chat interfaces. Evidence stored as screenshots or unstructured text. No GitHub Actions workflows present.

Stage 2 — Structured: Process is documented and role-assigned, but not automated. A defined sequence exists (agents commit before LLM synthesis; LLM synthesis before Human Score; Human Score before tag). Files follow a naming convention. DECISION.md records architectural choices. Automation is aspirational rather than operational.

Stage 3 — Automated: At least one GitHub Actions workflow fires on a scheduled trigger without manual intervention, commits a timestamped output file, and produces verifiable evidence of execution. Data collection is decoupled from human action. The pipeline can run during a study break.

Stage 4 — Intelligent: Multiple workflows operate in a coordinated cascade. CI validation checks agent output quality on every commit. API calls replace manual LLM interaction. Calibration scoring is automated and cumulative. The system monitors its own output quality and feeds results back into the next sprint's planning.

Each team's maturity was assessed at each sprint boundary using observable GitHub evidence, not self-report.

3.4 Analytical Approach

This study employs a multi-case interpretive analysis (Yin, 2013) supplemented by quantitative workflow metrics. For each team, the following metrics were computed:

  • Sprint delivery rate: Proportion of sprints with correct-format tag cut before deadline
  • Workflow success rate: Ratio of successful to total GitHub Actions runs
  • Automation onset sprint: Sprint number at which first successful scheduled workflow run occurred
  • Calibration coverage: Number of prediction files with subsequent delta scoring
  • Documentation depth: Count of sprint artefacts (goals, retrospectives, standups, acceptance criteria) committed per sprint

Anonymisation protocol: teams are identified as Team A through Team E in published findings. Repository URLs are available to reviewers on request but will not appear in the published paper.

4. The PRISM Framework — A Proposed Model for AI-Augmented Design Thinking Pedagogy

4.1 Overview

The PRISM Framework (Pipeline Readiness Index for Student Market Intelligence) proposes that effective AI-augmented design thinking education requires four integrated components operating simultaneously:

P — Pipeline Architecture: A four-layer technical structure (data collection → agent processing → LLM synthesis → sealing and calibration) that maps directly onto the Scrum sprint cycle.

R — Role Mapping: Ten defined roles, each corresponding to a professional function in industry, that distribute pipeline ownership across team members and eliminate single points of failure.

I — Instructor Infrastructure: A monitoring system that gives the educator real-time visibility into all student pipelines — commit activity, workflow status, tag compliance, PR activity — enabling evidence-based intervention before deadlines rather than after.

S — Scrum Cadence: A weekly sprint cycle with mandatory artefacts (DECISION.md, sprint goal, acceptance criteria, retrospective) that structures the team's improvement trajectory across eight weeks.

M — Maturity Measurement: The four-stage automation maturity rubric that provides objective, artefact-based assessment criteria independent of self-report.

4.2 The Four-Layer Pipeline Architecture

The pipeline architecture was not prescribed. It emerged from team practice and was identified through analysis of committed workflows, scripts, and output files. The following four-layer model describes what every effective team built, in varying degrees of automation maturity:

Layer 1 — Data Collection. A scheduled GitHub Actions workflow fetches market data after the US Friday close (Saturday morning SGT, approximately 04:00 UTC+8). The script commits a timestamped JSON file containing instrument closes for SPX, NDX, IWM, and sector ETFs. The timestamp is embedded in both the filename and the file content, creating an auditable record that data was collected before analysis began.

Layer 2 — Agent Processing. Three analysis scripts — Almanac (historical patterns and seasonality), Macro (economic calendar and monetary policy), Technical (price levels, trend direction, momentum) — consume the raw data file and produce structured Markdown analysis documents. Each agent commits independently. The commit order is enforced by workflow dependency: agents must commit before the LLM synthesis step executes.

Layer 3 — LLM Synthesis. An operator script sends identical prompts to multiple AI models via API (OpenRouter, OpenAI, Gemini, DeepSeek). Responses are saved as structured files in the repository. A comparison table is generated programmatically from those files. The Human Score lead reviews the comparison table and applies a Wild Card override — a sourced, non-obvious insight that the models missed — before committing the final prediction file.

Layer 4 — Sealing and Calibration. A release tag in the exact format vWXX is cut before the Sunday midnight deadline, creating the immutable sprint record. A calibration script scores the previous sprint's prediction against that week's actual market closes, producing a delta score (direction correct: Y/N; error percentage). Cumulative accuracy is tracked across all sprints.

4.3 The Educator Infrastructure Model

The educator infrastructure consists of three components deployed as a parallel technical system:

Monitor workflow: A Python script running every 30 minutes via GitHub Actions in a separate instructor repository checks all five student repos for new commits, workflow failures, tag creation, and open pull requests. Changes trigger immediate Telegram alerts to the instructor.

Live dashboard: A PIN-protected HTML page served via Cloudflare Pages displays real-time team status, delta indicators (change since last baseline), and direct links to GitHub repositories. Data is embedded at page build time and refreshed via GitHub API calls on each page load and every two minutes automatically.

Alert taxonomy: Alerts are categorised as: (1) new commit detected, (2) workflow failure, (3) workflow recovery, (4) new tag cut (with format validation — vWXX correct vs WXX or v1.0.0 incorrect), and (5) new pull request opened.

This infrastructure gives the instructor a complete picture of all team activities before each Monday class, enabling prepared, specific, evidence-based feedback rather than reactive assessment.

5. Results — Five Team Case Studies

5.1 Team A — Full-Stack Automation with CI Validation (PRISM Stage 4)

Repository metrics: 95 commits across sprints; 4 release tags (vW22–vW25, all correct format); 14 sprint documents; 100 workflow runs (75 success, 24 failure); first successful scheduled run: 26 June 2026.

Team A built the most architecturally complete pipeline in the cohort. Their backend/ directory contains a Python package structured as a professional software project: a pyproject.toml dependency manifest, a uv.lock lockfile, a pipeline.toml configuration file that toggles pipeline stages on and off, a run_pipeline.py entry point, and a BaseAgent abstract class with typed output schemas.

The schema design is particularly noteworthy. Rather than committing free-form Markdown, Team A defined Python dataclasses for every agent output type — TechnicalOutput, AlmanacOutput, MacroOutput, LLMOutput, EvidenceOutput — using Enum types for constrained fields (Bias.BULLISH, Confidence.HIGH, Regime.UNCERTAIN). This enforced structural consistency across all agent outputs from Sprint 2 onwards, eliminating the format drift that affected other teams.

@dataclass
class TechnicalOutput:
    prediction_date: date
    instruments: dict[str, InstrumentTechnical]  # keys: "SPX", "NDX", "IWM"
    agent_type: str = "technical"

@dataclass
class InstrumentTechnical:
    last_close: float
    ema_8: float
    ema_21: float
    trend_bias: Bias
    key_support: float
    key_resistance: float
    confidence: Confidence

Team A also implemented two CI workflows that run on path-filtered pushes, catching errors before they reach the weekly pipeline:

# agent-checks.yml — validates format on every push to agent output folders
on:
  push:
    paths:
      - "data/macro/**"
      - "data/almanac/**"
      - "data/technical/**"
      - "data/llm/**"

The pipeline.toml configuration file enabled the team to disable individual stages during testing without modifying code:

[pipeline]
prediction_date = "2026-06-08"  # W24

[stages]
almanac   = true
technical = true
macro     = true
evidence  = true

[llm]
models = ["nemotron", "gptoss", "gemma", "laguna"]

Their Sprint 5 prediction (filed Sunday 21 June 2026) demonstrated mature multi-LLM synthesis: four models (Nemotron, GPT-OSS, Gemma, Laguna via OpenRouter API) returned a consensus Neutral-Bullish regime for SPX, with a predicted range of -1.0% to +1.5%. The prediction file was sealed under tag vW25 at 16:20 UTC. Actual outcome: SPX +1.7% — within the predicted range, direction correct.

Sprint 5 retrospective documented authentic team reflection: "The team successfully progressed toward the Sprint 5 goal of achieving a more automated pipeline. Multiple project components were completed, including the Delta Engine, Evidence Agent, Human Score review, testing activities, and frontend development. Automation continued to reduce manual work and improved overall workflow efficiency."

PRISM classification: Stage 4 — Intelligent. CI validation, typed schemas, multi-model API calls, automated PR creation, 14 sprint artefacts.

Key learning for educators: Team A's DECISION.md documents the reasoning behind agents-first architecture: "The agents sit at the very beginning of the pipeline. Any errors or inconsistencies introduced at that stage propagate forward and compound, so getting them right is the highest-priority investment." This document represents exactly the kind of architectural reasoning that design thinking education should cultivate — and that is invisible to graders who evaluate only the final output.

5.2 Team B — Cascade Workflow Engineering with Automated Calibration (PRISM Stage 4)

Repository metrics: 100+ commits; 4 release tags (vW22–vW25, all correct format); 57 workflow runs (42 success, 15 failure); first successful scheduled run: 27 June 2026; W27 snapshot auto-collected during study break.

Team B built the most sophisticated workflow orchestration in the cohort, despite having a less architecturally clean code structure than Team A. Their three-workflow cascade represents a genuine industry pattern:

Workflow 1 — Automated Cross-Asset Market Audit Pipeline (cron: 17 22 5) collects multi-asset market data, generates SPX/NDX/IWM charts as PNG files, and commits market_snapshot_WXX.json.

Workflow 2 — Automated Technical Agent (cron: 22 14 0) reads the market snapshot and runs r5_technical_automation.py, which calculates EMAs, identifies pivot-based support and resistance levels, and commits technical_agent-WXX.md.

Workflow 3 — R10 Calibration Automation triggers via workflow_run on completion of Workflow 1, automatically scoring the previous week's prediction:

on:
  workflow_run:
    workflows:
      - "Automated Cross-Asset Market Audit Pipeline"
    types:
      - completed

The workflow_run trigger is one of the most powerful and underused patterns in GitHub Actions. It creates a self-driving pipeline chain without polling delays or manual triggers. When Workflow 1 completes, Workflow 3 fires automatically — regardless of whether any human has touched a keyboard.

Team B's workflow includes a dynamic week resolution step that automatically calculates both the market week and the prior actuals week from the system date:

WEEK_NUM=$(date +%V)
M_WEEK="W${WEEK_NUM}"
PREV_NUM=$((10#$WEEK_NUM - 1))
A_WEEK="W${PREV_NUM}"
echo "MARKET_WEEK=$M_WEEK" >> $GITHUB_ENV
echo "ACTUAL_WEEK=$A_WEEK" >> $GITHUB_ENV

This eliminates the need to manually update week numbers in configuration files — a common source of error in teams operating at Stage 2.

Evidence of automation maturity during the study break: Team B's pipeline auto-collected W27 data on 4 July 2026 (US Independence Day holiday), committing market_snapshot_W27.json with generation timestamp 2026-07-04 08:13:31. The fact that this ran on a public holiday — when markets were closed and no student was working — demonstrates that the pipeline was operating as genuine infrastructure rather than as scripted manual work.

Their calibration script (r10_calibration_automation.py, 17,860 bytes) covers all 11 S&P 500 sector ETFs in addition to the three core indices, with alias matching:

TARGETS = {
    "XLK": {"display": "XLK Technology",
             "actual_aliases": ["XLK Technology", "Technology", "XLK"]},
    "XLF": {"display": "XLF Financials",
             "actual_aliases": ["XLF Financials", "Financials", "XLF"]},
    # ... all 11 sectors
}

PRISM classification: Stage 4 — Intelligent. Workflow cascade, dynamic week resolution, automated calibration for 14 instruments.

5.3 Team C — Clean Data Collection with Minimal Viable Automation (PRISM Stage 3)

Repository metrics: 100+ commits; 5 tags (vW22–vW25 correct, plus erroneous W25); 3 workflow runs (all successful, 100% success rate); clean folder structure (vW22/, vW23/, vW24/, vW25/).

Team C's contribution to the cohort's collective knowledge is the cleanest data collection script. Their fetch_friday_closes.py is 47 lines of code that do exactly one thing correctly:

import json, os
from datetime import datetime
from zoneinfo import ZoneInfo
import yfinance as yf

SYMBOLS = {"SPX": "^GSPC", "NDX": "^NDX", "IWM": "IWM"}

sgt_now = datetime.now(ZoneInfo("Asia/Singapore"))
timestamp = sgt_now.strftime("%Y-%m-%d_%H-%M-%S_SGT")

output_dir = "data/market_closes"
os.makedirs(output_dir, exist_ok=True)

results = {
    "fetch_time_sgt": sgt_now.isoformat(),
    "note": "Automated GitHub Actions fetch for Friday market closes.",
    "symbols": {}
}

for name, ticker in SYMBOLS.items():
    data = yf.Ticker(ticker).history(period="5d")
    if data.empty:
        results["symbols"][name] = {"ticker": ticker, "error": "No data returned"}
        continue
    last_row = data.tail(1).iloc[0]
    last_date = str(data.tail(1).index[0].date())
    results["symbols"][name] = {
        "ticker": ticker,
        "last_trading_date": last_date,
        "close": round(float(last_row["Close"]), 2)
    }

output_file = f"{output_dir}/friday_closes_{timestamp}.json"
with open(output_file, "w", encoding="utf-8") as f:
    json.dump(results, f, indent=2)

Three design decisions in this script merit specific attention in any educator-facing curriculum:

SGT-anchored timestamps. Using ZoneInfo("Asia/Singapore") rather than UTC or local system time ensures that output filenames and embedded timestamps are unambiguous regardless of where the GitHub Actions runner executes. The committed filename friday_closes_2026-07-03_23-47-45_SGT.json is self-documenting.

Per-ticker error handling. The if data.empty: branch means a failed ticker fetch does not crash the entire script — it commits a partial result with an error field. Other teams' scripts failed silently on ticker errors, producing empty commits with no diagnostic information.

os.makedirs(output_dir, exist_ok=True). This single line prevents the most common first-run failure in CI environments: the assumption that a folder already exists. It is the difference between a script that works on the developer's laptop and one that works reliably in GitHub Actions.

Team C's DECISION.md documents their reasoning for data-collection-first automation: "It was the most repetitive manual task. Last week, one team member had to open multiple websites, copy values, take screenshots, and commit the evidence to Github. It provides immediate value. Automating data collection reduces manual workload, speeds up preparation, and gives the team a consistent starting point for analysis."

This is authentic design thinking: identify the highest-friction step, automate it first, and build from there. The fact that it was documented in a DECISION.md committed to the repository — rather than mentioned in passing in a presentation — is what makes it pedagogically durable.

PRISM classification: Stage 3 — Automated. Single scheduled workflow, 100% success rate, clean output structure. Limited to data collection layer; agent processing and LLM synthesis remain manual.

Key finding: Team C demonstrates that Stage 3 automation — a single working scheduled workflow — is achievable in Sprint 3 by a team with no prior CI/CD experience, provided the automation scope is deliberately limited to the highest-friction task.

5.4 Team D — Modular Collector Architecture with Dual-Schedule Design (PRISM Stage 3–4 Transition)

Repository metrics: 100+ commits; 3 tags (vW24–vW25 correct, W23 incorrect); 14 workflow runs (12 success, 2 failure); 86% workflow success rate; first successful scheduled run: 13 June 2026 — earliest in the cohort.

Team D (PRISM: Predictive Reasoning and Intelligence Synthesis for Markets) built the most modular collector architecture. Their prism/src/collectors/ folder contains four independent Python modules:

prism/src/collectors/
  almanac_collector.py     # Calendar flags, sector seasonality, all 11 ETFs
  macro_collector.py       # Economic indicators, Fed data
  technical_collector.py   # Price history, EMA calculation, support/resistance
  seasonality_collector.py # Monthly/weekly pattern analysis

Each collector module is self-contained: its own docstring specifying input sources and output schema, its own path resolution using Path(__file__).resolve().parents[N], and its own save_json() utility function. The technical_collector.py already covers the full sector ETF universe — all 11 S&P 500 sectors — from Sprint 5:

TECHNICAL_TICKERS = {
    "SPX": {"ticker": "^GSPC", "name": "S&P 500 Index"},
    "NDX": {"ticker": "^NDX",  "name": "Nasdaq 100 Index"},
    "IWM": {"ticker": "IWM",   "name": "Russell 2000 ETF Proxy"}
}

Team D's most architecturally significant contribution is the dual-schedule workflow:

on:
  schedule:
    # Monday 8:00 AM Singapore time — agent collectors
    - cron: "0 0 * * 1"
    # Saturday 6:00 AM Singapore time — actuals after US Friday close
    - cron: "0 22 * * 5"

Combined with conditional step execution based on which schedule triggered:

- name: Run R6 actuals collector
  if: github.event_name == 'workflow_dispatch' ||
      github.event.schedule == '0 22 * * 5'
  run: python prism/src/data_collector.py

- name: Run technical collector
  if: github.event_name == 'workflow_dispatch' ||
      github.event.schedule == '0 0 * * 1'
  run: python prism/src/collectors/technical_collector.py

This pattern — one workflow, two cron triggers, conditional step execution — is an industry-standard approach to multi-phase batch processing. It eliminates the need for separate workflow files for each phase, reducing maintenance overhead and keeping the Actions log readable.

Evidence of genuine autonomy: Team D recorded three automatic commits from github-actions[bot] during the study break (24 June, 26 June, 29 June, and 6 July 2026), including the most recent commit as of the study period: chore: update Prism data 2026-07-06. Their prism/data/output.json from 6 July contained complete market data (SPX: 7483.24, NDX: 29329.21, IWM: 297.58, VIX: 16.15, BTC: 62962.51) committed automatically at 22:57 SGT — without any student being present.

PRISM classification: Stage 3–4 Transition. Four modular collectors, dual-schedule design, study break auto-collection, but LLM API calls and calibration automation not yet operational.

5.5 Team E — Automation Theatre and the Infrastructure-Without-Pipeline Anti-Pattern (PRISM Stage 1–2)

Repository metrics: 97 commits; 4 tags (all incorrect format: v1.0.0, W23, W24, W25); 16 workflow runs (10 success, 6 failure, 62.5% success rate); no auto-collected market data files committed.

Team E represents the most instructive failure case in the cohort — instructive precisely because the failure is non-obvious. Their repository contains genuine engineering ambition: a Next.js frontend, an Express.js backend deployed to Render, a multi-file Python data collection suite (fetch_yfinance.py, fetch_finviz.py, run_weekly_fetch.py, macro_agent.py), and three GitHub Actions workflows.

Yet their pipeline produced no automatically committed market data during the study break. Their workflow success rate (62.5%) is the lowest in the cohort. Their tags have been in the wrong format since Sprint 1 — four sprints of W23, W24, W25 rather than vW23, vW24, vW25.

The root cause is what this paper terms the automation theatre anti-pattern: the construction of automation infrastructure that appears operational without producing verifiable automated output. Specifically:

  1. Their Weekly Team File Consolidation workflow (cron: every Sunday 15:00 UTC) moves files from an incoming/ folder to an evidence/Week X/ directory. This produces a green checkmark in the Actions log but performs no market data collection.
  1. Their Run Macro Agent Remote workflow fires on the macro-update branch, not on main — meaning it only runs when a developer manually pushes to a specific branch, not on schedule.
  1. Their Python data collection scripts (fetch_yfinance.py, fetch_finviz.py) exist in the TradeKyaMal/scripts/ folder but are not called by any workflow.

The distinction between automation and automation theatre is visible only by examining what the workflow produces, not whether it runs. A green Actions badge on a file-moving workflow is not a pipeline. The test is: did github-actions[bot] commit a timestamped market data file during the study break? For Team E, the answer is no.

PRISM classification: Stage 1–2 Boundary. Structured process, documented roles, genuine engineering effort, but no functional automated pipeline output.

Key finding for educators: Automation theatre is identifiable through the GitHub API by checking for commits from github-actions[bot] (genuine automation) versus commits from named authors (human-triggered work). This distinction is invisible in traditional assessment but trivially observable in the instructor dashboard.

6.1 Automation Maturity Progression Across Sprints

Two findings are immediately apparent. First, the number of active workflows is not a reliable indicator of automation maturity: Team E has three active workflows and the lowest maturity score; Team C has one workflow and a higher maturity score. Second, the study break auto-commit data provides the strongest single indicator of functional automation — a pipeline either ran without human intervention or it did not.

6.2 Sprint Delivery Consistency

Teams at Stage 3 and above (A, B, C, D) achieved correct-format release tags on time for all five active sprints. Team E's tag format was incorrect in all four sprints where tags were created. This finding supports the hypothesis that automation maturity and process discipline are correlated — teams who invest in automating repetitive tasks also invest in documenting and enforcing process conventions.

The tag format error in Team E is particularly instructive. W25 vs vW25 is a two-character difference. It persisted across four sprints despite repeated correction in class feedback. Its persistence is not explained by ignorance — Team E knew the correct format. It is explained by the absence of a DoD checklist item: when "correct tag format" is not an explicit, written, checked item in the sprint Definition of Done, it is omitted under time pressure. This finding replicates the broader Scrum literature on the necessity of explicit DoD artefacts [CITATION — VERIFY: e.g. Schwaber, K., & Sutherland, J. (2020). The Scrum Guide, on Definition of Done; or empirical DoD-adherence studies].

6.3 The Educator Infrastructure Effect

The instructor monitoring system described in Section 4.3 provided measurable pedagogical benefit in three ways.

First, it enabled pre-class preparation that was qualitatively different from traditional assessment. Rather than reviewing submitted files on the morning of class, the instructor had been receiving Telegram alerts throughout the week — pipeline failure at 03:00 on Thursday, new tag cut at 23:47 on Sunday, five open pull requests in Team A. Monday class preparation could focus on specific, evidenced observations rather than general encouragement.

Second, it enabled team-differentiated private feedback. Teams received private channel messages after Sprint 5 that cited specific repo evidence: "Your fetch ran on Thursday night and committed real data. Show me that file. What were the SPX and NDX closes?" This level of specificity is only possible with automated monitoring.

Third, it revealed the automation theatre anti-pattern before it became entrenched. Without the dashboard, Team E's three green workflow badges would have appeared equivalent to Teams A and B. The dashboard's filter — github-actions[bot] commits vs named-author commits — surfaced the distinction immediately.

6.4 Calibration Accuracy as a Learning Metric

Teams A and B both filed complete prediction files with direction, range, and confidence ratings for each sprint. Team A's Sprint 5 prediction (SPX: FLAT-UP, -1.0% to +1.5%, MEDIUM confidence) was correct: actual SPX moved +1.7%. Team B's automated W27 market snapshot, committed during the study break, demonstrated that their pipeline was collecting data that would enable accurate calibration even without student involvement.

The calibration metric — did the direction prediction match the actual outcome? — provides an objective, non-gameable assessment criterion for prediction quality. Unlike rubrics based on presentation quality or documentation completeness, direction accuracy either matches or it does not. Cumulative accuracy over six sprints provides a meaningful signal of whether AI synthesis, guided by structured agents and a Human Score override, produces better predictions over time — the central question of the entire course.

6.5 Limitations

This study has five limitations that future research should address.

Single cohort: All data comes from one trimester (TR2 2026) with 56 students. Results may not generalise to larger cohorts, different disciplines, or different course durations.

Self-selection of course design: The course was designed by the researcher. This introduces potential design bias toward findings that support the framework. Independent replication in a different institution with a different educator is needed.

GitHub API sampling: Workflow run history is limited to the most recent 100 runs per repository. Teams with very high activity levels may have older sprint data truncated.

Prediction accuracy causality: The correlation between automation maturity and prediction accuracy observed in Teams A and B may reflect team competence generally rather than automation specifically. A controlled study with random assignment of automation support would be required to establish causality.

Anonymisation trade-off: By anonymising teams A–E, the paper sacrifices the ability to link findings to the specific GitHub repositories that reviewers could independently verify. Reviewer access to repositories will be provided on request.

7. Implications for Practice

7.1 For Educators

Three implications follow directly from these findings.

Deploy a monitor before Sprint 1, not after. The instructor monitoring system is most valuable when it establishes baseline awareness from the first sprint. Teams whose early tag format errors are caught and corrected in Sprint 1 do not carry them through Sprint 5. Monitoring set up after Sprint 3 has limited corrective impact.

Mandate DECISION.md from Sprint 1. Team A's DECISION.md entries document architectural reasoning that is invisible in the final product. "The agents sit at the very beginning of the pipeline. Any errors or inconsistencies introduced at that stage propagate forward and compound." This is the intellectual output that design thinking education exists to produce. Without the DECISION.md requirement, it is not committed, not evaluated, and not credited.

Distinguish automation from automation theatre in assessment. The presence of GitHub Actions workflows is not evidence of automation. The presence of timestamped github-actions[bot] commits is. Assessment rubrics should require specific automated output files as evidence, not workflow badge screenshots.

7.2 For Curriculum Designers

The PRISM four-layer pipeline architecture is proposed as a replicable template for AI-augmented design thinking courses in any domain that involves regular data collection, multi-agent analysis, and prediction or recommendation delivery. The domain (market intelligence) is incidental. The structure (collect → analyse → synthesise → seal) is generalisable.

The ten-role system maps cleanly onto professional functions across technology, consulting, and finance. Students who complete six sprints as LLM Operator (R8) have made programmatic API calls to at least two AI models, saved structured responses as version-controlled files, and built comparison tables from those files. This is a demonstrable, verifiable skill with direct workforce relevance.

7.3 For Researchers

The GitHub API provides educational researchers with access to a class of objective, timestamped, non-self-reported data that is rare in education research. Commit timestamps, workflow run outcomes, tag creation records, and pull request lifecycles are all accessible via public API and require no ethical approval beyond confirmation that repositories are public.

Future research should examine: (1) automation maturity trajectories across multiple cohorts; (2) the relationship between retrospective quality and next-sprint improvement; (3) whether the PRISM model generalises to non-market domains; and (4) the effect of educator monitoring visibility on student intrinsic motivation versus compliance behaviour.

8. Conclusion

This paper presented an empirical study of AI-augmented Scrum pedagogy in a live design thinking course, drawing on 492 GitHub commits, 190 workflow run logs, and 20 prediction files from five student teams across six active sprints. The PRISM Automation Maturity Model provides a four-stage rubric — Manual, Structured, Automated, Intelligent — that is measurable through observable GitHub artefacts and independent of self-report.

Five team case studies demonstrated the full spectrum from Stage 4 automation (full-stack pipeline with typed schemas, CI validation, and multi-model API synthesis) to Stage 1–2 automation theatre (three active workflows producing no functional automated output). The most important single indicator of functional automation — the presence of commits from github-actions[bot] during a period of no student activity — distinguishes real pipelines from decorated repositories.

The Educator Infrastructure Model — GitHub Actions monitor, Telegram alerts, PIN-protected live dashboard — demonstrated that real-time instructor visibility into student pipeline health enables qualitatively different pedagogical intervention: specific, evidenced, pre-class preparation that addresses individual team failures rather than cohort-wide generalities.

The long-term significance of this work lies not in the market intelligence domain but in the framework it demonstrates: that design thinking education, when structured around repeatable automated delivery under Scrum constraints, can produce workforce-ready skills in DevOps, AI/ML engineering, data analysis, and calibrated judgment — skills measured not by rubric scores but by whether the machine ran while the student was sleeping.

Appendix B — Sample Data Collection Script (Team C fetch_friday_closes.py)

import json, os
from datetime import datetime
from zoneinfo import ZoneInfo
import yfinance as yf

SYMBOLS = {"SPX": "^GSPC", "NDX": "^NDX", "IWM": "IWM"}

sgt_now = datetime.now(ZoneInfo("Asia/Singapore"))
timestamp = sgt_now.strftime("%Y-%m-%d_%H-%M-%S_SGT")
output_dir = "data/market_closes"
os.makedirs(output_dir, exist_ok=True)

results = {
    "fetch_time_sgt": sgt_now.isoformat(),
    "note": "Automated GitHub Actions fetch for Friday market closes.",
    "symbols": {}
}

for name, ticker in SYMBOLS.items():
    data = yf.Ticker(ticker).history(period="5d")
    if data.empty:
        results["symbols"][name] = {"ticker": ticker, "error": "No data returned"}
        continue
    last_row = data.tail(1).iloc[0]
    results["symbols"][name] = {
        "ticker": ticker,
        "last_trading_date": str(data.tail(1).index[0].date()),
        "close": round(float(last_row["Close"]), 2)
    }

output_file = f"{output_dir}/friday_closes_{timestamp}.json"
with open(output_file, "w", encoding="utf-8") as f:
    json.dump(results, f, indent=2)
