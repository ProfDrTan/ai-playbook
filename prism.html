What Each Team Contributed to the Collective Knowledge

Every team produced something the others didn't. This is what gets folded into Team Ultra.

Team 1 · Georgii (sinder38)
Professional Software Architecture
Abstract base class BaseAgent with typed output schemas using Python dataclasses and Enums (Bias.BULLISH, Confidence.HIGH). pipeline.toml config toggles stages on/off without code changes. CI validation on every push to agent output folders. GitHub Actions environment gate requires human approval before production run.

Key insight: Typed schemas eliminate format drift across agents. Once defined in Sprint 2, every subsequent agent output is structurally consistent.
Team 2 · LuJunwen + The Bao
Workflow Cascade + SM Discipline
workflow_run trigger chains three workflows automatically — audit → technical → calibration — without polling or manual triggers. Dynamic week resolution (WEEK_NUM=$(date +%V)) eliminates manual config updates. R10 calibration covers all 11 S&P sectors with alias matching.

Key insight: The Bao's individual Wednesday DMs with per-role DoD checklists is the only documented mid-week check-in that matches the Scrum Master spec. This is the SM reference pattern.
Team 3 · Vince-1206
The Minimal Correct Fetch Script
47 lines. SGT-anchored timestamps via ZoneInfo("Asia/Singapore"). Per-ticker error handling that commits partial results rather than crashing. os.makedirs(exist_ok=True) for first-run safety. Self-documenting JSON with a "note" field explaining automation context. 100% workflow success rate across all sprints.

Key insight: Simplicity and correctness beat complexity and fragility. Teams 1 and 2 have more impressive pipelines — Team 3 has the most reliable data collection.
Team 4 · Arisa (wintwah)
The Automation Theatre Anti-Pattern
Four sprints of workflows that ran with green checkmarks but produced no substantive automated output. File consolidation masquerading as data automation. Scripts that existed but were never called by any workflow. Tag format wrong in every sprint.

Key insight: The most valuable teaching contribution in the cohort. Distinguishing a green Actions badge from a functioning pipeline is the single most important diagnostic skill for future students. Team 4 is the reference case for what not to do — and for what recovery looks like (Sprint 6: first genuine auto-commit).
Team 5 · ToHoangGia + Gong
Dual-Schedule Design + Modular Collectors
One workflow, two cron triggers, conditional step execution: if: github.event.schedule == '0 22 * * 5'. Monday fires agent collectors; Saturday fires actuals fetch. Four independent collector modules each with their own docstring, path resolution, and output spec. Most consistent auto-commits in cohort — 4 during study break, none missing.

Key insight: Separating the Monday agent cycle from the Saturday actuals fetch in one workflow is elegant infra design. No other team did this.

sgt_now = datetime.now(ZoneInfo("Asia/Singapore"))
results = {
    "fetch_time_sgt": sgt_now.isoformat(),
    "note": "Automated GitHub Actions fetch — Friday closes.",
    "symbols": {}
}
for name, ticker in SYMBOLS.items():
    data = yf.Ticker(ticker).history(period="5d")
    if data.empty:
        results["symbols"][name] = {"error": "No data returned"}
        continue
    results["symbols"][name] = {"close": round(float(data.tail(1).iloc[0]["Close"]), 2)}
Layer 2 — Scheduler
Team 5's dual-cron
Team 5
One workflow, two triggers, conditional execution. Monday fires agents. Saturday fires actuals. No separate workflow files to maintain. 
on:
  schedule:
    - cron: "0 0 * * 1"   # Mon 8am SGT — agent collectors
    - cron: "0 22 * * 5"  # Sat 6am SGT — actuals after US Friday close

steps:
  - name: Run agent collectors
    if: github.event.schedule == '0 0 * * 1'
    run: python collectors/run_all.py

  - name: Run actuals fetch
    if: github.event.schedule == '0 22 * * 5'
    run: python collectors/fetch_closes.py

Key Insights for Future Trimesters
The Green Badge Fallacy
A GitHub Actions workflow with a green checkmark is not evidence of a working pipeline. It is evidence that the workflow ran without crashing. The diagnostic question is: did github-actions[bot] commit a timestamped output file? If yes — automation. If no — the workflow moved files, ran tests, or did something cosmetic. Team 4 spent four sprints with green badges and no functional output. Teach this distinction in Sprint 1.
A Failed Pipeline That Tried More Is More Advanced Than a Successful Pipeline That Did Less
Team 1's pipeline "failed" in Sprint 7 because it attempted to call four LLM models simultaneously. Two succeeded. Two were rate-limited. Teams 3, 4, and 5 had "successful" pipelines that never made an LLM call at all. The failure revealed the ambition. The success concealed the absence of it. Assessment should reward what the pipeline attempted, not just whether the workflow returned exit code 0.
The Compounding Retrospective
Team 2's R10 calibration accuracy improved sprint-over-sprint because every retrospective produced one concrete change to the calibration script. By Sprint 7, their calibration covered all 11 S&P sectors automatically and fired via workflow_run without any human trigger. No single sprint decision was dramatic. The compound effect of 6 small improvements was. This is the core pedagogical argument for Scrum in an AI course.
