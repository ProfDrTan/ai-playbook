Changelog

Every entry below is added at the point of a confirmed repo change — not in advance.

2026-07-10 — COMMANDMENT
Thou shalt not build a Cloudflare Worker for the CP5640 Participation Tracker. Ever again.

It has already failed twice on this exact problem — once on the CP5640 tracker itself (~2 hours and significant token spend lost), and once before that on the research-methods site, for the identical reason. A Worker is new infrastructure with its own auth, deployment, and debugging surface, and it is not needed here.

The proven fix: use the GitHub API as the sole shared data store — read/write student participation data directly to a JSON file in the tracker repo via the GitHub API, fetching a fresh SHA immediately before every PUT. No Worker, no proxy, no new infrastructure. Test in sandbox first, verified across multiple devices, before it ever touches the live tracker again.

2026-07-09 — Learning: Prompt Format Root Cause
Problem observed: across every live run since Phase 3 went live, Nemotron and Laguna failed consistently — not with API errors, but with no JSON object found in response or empty response content (finish_reason='length'). Both are reasoning models: they write out a full chain-of-thought before their final answer. Asking them for "ONLY a JSON object" meant the JSON either never arrived within the token budget, or arrived buried inside prose that a strict json.loads() call rejected outright.

Root cause found by reading Team 1's actual code (see the Code Walkthrough, section 7): their multi_model_runner.py system prompt explicitly states "DO NOT OUTPUT JSON FORMAT" and instructs a plain KEY: value text format instead. Their parser in base_llm.py scans every line for a colon rather than parsing the whole response as one JSON blob — which degrades gracefully around a reasoning trace instead of breaking on it.

Fix applied to agents/llm/multi_model_runner.py:
  • Prompt changed from "respond with ONLY a JSON object" to a labeled DIRECTION: / RANGE_LOW: / RANGE_HIGH: / CONFIDENCE: / REASONING: plain-text format, matching Team 1's field-per-line convention
  • Added an explicit system message banning JSON output, adapted directly from Team 1's wording
  • extract_json() replaced with parse_fields() — scans lines for KEY: value pairs, tolerant of surrounding reasoning text, instead of requiring one parseable JSON object
  • Retry logic changed from "retry only on HTTP 429" to retry on any exception with exponential backoff, matching Team 1's 2 ** attempt pattern
Verified result (run immediately after the fix, log committed at data/logs/pipeline_20260709_164149.txt): Nemotron and Laguna succeeded on every call, zero JSON-parse errors. GPT-OSS and Gemma still hit 429 — that's the separate, already-diagnosed OpenRouter account rate-limit issue, unrelated to prompt format and not something this fix was meant to address. The new any-exception retry logic also caught a real transient miss live: Laguna missed RANGE_HIGH on IWM's first attempt and succeeded on the automatic retry.

Glossary

Plain-language definitions for terms that show up in this build log and its commit history.

OpenRouter Service
A single API that sits in front of dozens of different AI providers (OpenAI, Google, NVIDIA, Meta, and others). One API key lets the pipeline call several different models without separate accounts for each.
Free-tier model Concept
A model OpenRouter hosts at zero cost, subsidized by them. In exchange, it's rate-limited — a capped number of requests per minute/day — and can be slower or occasionally unavailable compared to a paid model.
Nemotron Model
An NVIDIA-built open-weight model. The variant used here is a "reasoning" model — it writes out its thinking before its final answer, which is why it needs a larger token budget than a model that answers directly.
GPT-OSS Model
OpenAI's open-weight model (separate from GPT-4/GPT-5 — "open-weight" means the model file itself is publicly downloadable, not just accessible via API).
Gemma Model
Google DeepMind's open-weight model family, distinct from Gemini (Google's flagship, closed model).
Laguna Model
An open-weight model from Poolside, a smaller AI lab. Included to match Team 1's original four-model panel; currently the least reliable of the four in testing.
HTTP 429 Error code
"Too Many Requests." The server understood the request fine but is refusing it because too many requests arrived too quickly. The fix is slowing down, not fixing a bug in the request itself.
Rate limit Concept
A cap a service imposes on how many requests you can make in a given time window, to protect their infrastructure. Free tiers almost always have tighter rate limits than paid tiers.
Trust weight This project
A per-model score (0–1) representing how accurate that model's predictions have historically been. Models that predict well get weighted more heavily in future synthesis; models that predict poorly get weighted less — but never to zero.
EMA (Exponential Moving Average) Method
A way of updating a running average that weights recent results more heavily than old ones, without needing to store the full history. Used here so a model's trust weight adapts as its recent accuracy changes.
Calibration This project
Comparing what the models predicted against what actually happened in the market, to score accuracy and update trust weights. This is the automated replacement for Team 1's manual weekly QA step.
Synthesis This project
Combining all four models' individual predictions into one final trust-weighted prediction, rather than just averaging them equally.
yfinance Library
A free Python library that pulls historical stock/index price data from Yahoo Finance. Used by the Technical and Macro agents — confirmed working reliably in every test run so far.
RSI (Relative Strength Index) Indicator
A technical indicator from 0–100 measuring how fast and how far a price has moved recently. Above 70 is typically read as "overbought," below 30 as "oversold."
GitHub Actions Infrastructure
GitHub's built-in automation runner — lets code execute on a schedule (like the weekly pipeline) without a server you have to maintain yourself.
workflow_dispatch GitHub Actions
A setting that lets a GitHub Actions workflow be triggered manually (via button or API call) in addition to its normal schedule — used here to test the pipeline without waiting for Monday.
Repo secret GitHub Actions
An encrypted value (like an API key) stored on a GitHub repo that workflows can use at runtime without the value ever appearing in the code itself.
Fine-grained PAT GitHub
A newer style of GitHub access token where permissions are broken into separate categories — Contents, Workflows, Secrets, Actions, etc. — each needing to be explicitly enabled, unlike older "classic" tokens which used broader all-or-nothing scopes.
SHA (in a Git context) Git
A unique fingerprint identifying the exact current version of a file. GitHub's API requires fetching this right before any edit, to make sure you're not overwriting someone else's more recent change.
